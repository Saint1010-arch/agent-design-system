## 一、OpenClaw 架构核心（官方文档精华）

### 1.1 文件即灵魂
- 记忆完全依赖 workspace 文件。模型只记住写入磁盘的内容。
- `SOUL.md`/`AGENTS.md`/`IDENTITY.md`/`USER.md`/`TOOLS.md`/`HEARTBEAT.md` 每轮注入上下文。
- `memory/*.md` 日文件**不自动注入**，靠 `memory_search`/`memory_get` 按需调取。
- **Token 预算**：每个注入文件消耗上下文窗口。`bootstrapMaxChars` 默认 20000/文件，总计 150000。

### 1.2 子 Agent 隔离规则
- 子 Agent **只注入 AGENTS.md + TOOLS.md**（不继承 SOUL/IDENTITY/USER/HEARTBEAT）。
- 默认无 session 工具。深度嵌套最多 5 层（推荐 2 层）。
- 每个父 Agent 最多 5 个活跃子 Agent，全局并发上限 8。

### 1.3 多 Agent 路由
- 每个 `agentId` 完全隔离（workspace + agentDir + sessions）。
- 通过 `bindings` 路由（peer → accountId → channel → default）。
- 工具权限、沙箱可按 Agent 定制。

### 1.4 Skill 系统
- Skill = 含 `SKILL.md` 的目录 + YAML frontmatter。
- 加载优先级：workspace > managed > bundled。
- 支持门控：`requires.bins`/`requires.env`/`requires.config`。
- 每个 Skill 约 97+字符的 token 开销。

### 1.5 记忆与压缩
- 自动压缩：接近上下文窗口时总结旧对话。压缩前有 memory flush。
- 向量搜索：BM25 + 向量混合搜索，支持 MMR 去重 + 时间衰减。
- 会话默认每天 4:00 AM 重置。

## 二、业界 Prompt 设计最佳实践

### 2.1 Anthropic 核心原则
- **清晰直接**：把 AI 当缺乏上下文的新员工。
- **给理由**：解释"为什么"让模型泛化。
- **示例驱动**：3-5 个多样化示例，`<example>` 标签包裹。
- **XML 结构化**：分隔指令、上下文、输入。
- **角色设定**：一句话角色描述就能显著聚焦行为。

### 2.2 OpenAI 四维人格模型
| 类型 | 适用 | 特征 |
|:---|:---|:---|
| **Professional** | 企业/法律/金融 | 正式、精确、交易感 |
| **Efficient** | 代码/自动化 | 极简、直达、不加码 |
| **Fact-based** | Debug/评审/风控 | 坦率、纠正、基于证据 |
| **Conversational** | 客服/教育 | 温暖、鼓励、适度随意 |

**关键**：人格不覆盖任务特定输出格式；从最小人格开始通过 eval 演进。

### 2.3 LangChain 多 Agent 模式
| 模式 | 机制 | OpenClaw 对应 |
|:---|:---|:---|
| **Subagents** | 主 Agent 把子 Agent 当工具调 | `sessions_spawn` |
| **Skills** | 按需加载专业 prompt | Skill 系统 |
| **Handoffs** | Agent 间传递控制权 | `sessions_send` + bindings |
| **Router** | 路由器分流，不参与执行 | bindings 路由 |

| 模型 | 来源 | 做法 | 适用角色 |
|:---|:---|:---|:---|
| **第一性原理** | Musk/亚里士多德 | 拆到基本事实，从零重构 | 战略/架构/产品 |

## 四、360 行业背景与产品级标准

### 4.1 龙虾生态定位
- OpenClaw 2026年初开源，GitHub 星标 180K+，国内各大云厂商适配。
- 360 纳米AI事业部参与从 0→1 推出 360 自己的龙虾产品。
- 产出物需达到**技术部上架审核标准**——不是个人玩具，是官方产品。

### 4.2 产品级 Agent 上架标准（综合推导）
1. **结构完整性**：五层架构无缺失
2. **可移植性**：另一个人读了能直接接手
3. **可测试性**：每条行为规则可判断真假
4. **安全合规**：明确的数据安全边界、操作权限
5. **差异化**：与现有 Agent 不重复、有独特价值
6. **Token 效率**：核心配置精炼，不浪费上下文
7. **文档规范**：Markdown 格式、标题层级、命名一致
8. **Eval 就绪**：附带可用于质量评估的测试用例/场景

### 4.3 企业级 Agent 评估框架（UiPath/AWS/Kore.ai 综合）
- **评估驱动开发**：企业 AI Agent 不是靠 demo 定义的，是靠 evaluation 定义的。
- **持续评审节奏**：每周/双周跨职能团队 review Agent 行为——什么好用、什么出错、怎么调整。
- **数据域映射**：CDO + CISO 联合确定哪些数据域可以让 Agent 自主决策、哪些需要改进、哪些是硬边界。
- **模拟测试**：用 AI 模拟数百个场景和多样化用户角色，监控 Agent 在每一步的反应。

## 五、审美工程——弥合 AI "模拟理解"与"真正理解"的 Gap

### 5.1 问题本质

AI 的"理解"是统计模式匹配，不是基于体验的直觉。它能流畅输出"用户体验很重要"，但不一定真正理解什么是好的用户体验。这个 gap 在简单任务中不明显，但在**需要判断力、品味、专业深度**的场景中会暴露。

### 5.2 工程化弥合方案

| 手段 | 做法 | 解决的问题 |
|:---|:---|:---|
| **定义锚定** | 对核心概念写精确定义（≤2句），作为讨论锚点 | 防止"用对了词但理解偏了" |
| **行业顶级对标** | 找到该领域最顶级的人/公司的具体做法 | 把"好"从抽象形容词变成可对照的具体标准 |
| **反例检验** | 对每个判断主动构造反例 | 防止确认偏误 |
| **来源追溯** | 重要观点标注出处和时间 | 区分"我生成的"和"有权威来源的" |
| **置信度标注** | 🟢确定 / 🟡有依据推断 / 🔴推测 | 防止 AI 把猜测当确定性输出 |
| **能力圈声明** | 超出可靠判断范围时明确说 | 防止编造看似合理的答案 |
| **交叉验证** | 同一结论从≥2个独立角度验证 | 防止单一来源的偏见 |
| **版本意识** | 标注信息时间，区分过时与当下 | 防止用 2020 年的方法做 2026 年的事 |

### 5.3 "流畅的空话"检测法

AI 最危险的输出不是明显的错误，而是**看起来正确、读起来流畅、但实际没有信息增量**的内容。

检测方法：
1. **删除测试**：删掉这句话后，读者损失了什么信息？如果答案是"没有"，删掉。
2. **替换测试**：把这段话中的专业术语替换成其他领域的术语，读起来还通顺吗？如果通顺，说明它只是模板化输出，没有真正的领域理解。
3. **追问测试**：对这个结论追问"为什么？依据是什么？"，如果只能回答"通常来说"、"一般认为"，说明深度不够。

### 5.4 反压缩原则（来源：Mr. Ranedeer "Do not compress your responses"）

AI 的默认倾向是"总结"——压缩信息、删掉细节、给出框架性回答。这恰好是"流畅的空话"的温床。

原则：**不要为了精简而牺牲信息量。** 宁可写长但每句有信息增量，不要写短但只剩空洞的框架。"精炼"≠"简短"，精炼是每个字都有信息密度，简短可能只是偷懒。

### 5.5 虚假表现检测（来源：Human 3.0 "False Transformation"）

不只定义"什么是好的"，更要精确定义"什么看起来好但其实不好"。

每个核心能力都应该有"虚假表现清单"——看起来做到了但实际没做到的具体行为。例如：
- **虚假的"用户思维"**：嘴上说"以用户为中心"，但做决策时从未引用过一条用户数据
- **虚假的"战略思维"**：能说出 SWOT、波特五力，但无法判断当前场景该用哪个
- **虚假的"技术深度"**：堆砌术语但无法解释术语之间的因果关系

### 5.6 跨领域因果链（来源：Human 3.0 "Cross-Quadrant Problem Chains"）

顶级专家看的是系统动力学，不是孤立症状。表面问题的根因往往藏在另一个领域：
- 产品功能堆砌 → 根因可能是缺乏用户调研方法论
- 团队执行力差 → 根因可能是目标定义模糊
- 技术债务累积 → 根因可能是产品路线图频繁变更

在设计子龙虾时，为复杂角色显式标注"本领域常见问题的跨领域根因"。

### 5.7 适应性交互（来源：Human 3.0 "Adaptive Interview"）

同一个建议对不同水平的用户有不同价值和不同风险。优秀的 Agent 应该根据 User 的表现动态调节：
- **语言复杂度**：User 展示出高认知水平时用更精准的术语，反之则用更直白的解释
- **挑战度**：User 能接住质疑时加大质疑力度，反之则先给支撑再引导
- **建议激进度**：User 基础扎实时可以给高风险高回报建议，基础薄弱时给稳健路径

## 六、训练心法总结

### 子龙虾设计黄金三角
```
        [角色认同]
         /       \
  [方法论注入] — [行为边界]
```

### Prompt 工程十诫
1. 角色 > 指令
2. 禁止清单 > 鼓励清单
3. 示例 > 描述
4. 结构 > 自由发挥
5. 理由 > 命令
6. 具体 > 泛泛
7. 分层 > 堆砌
8. 最小权限
9. 可测试
10. 人格是杠杆
