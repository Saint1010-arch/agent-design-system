# 🦞 Agent 生产手册 — 标准化 SOP

_本手册定义了从"一个模糊的需求"到"一个完整的 Agent"的标准化生产流程。_

---

## 🔄 生产全流程

```
需求输入 → 需求澄清 → 角色设计 → 配置生成 → 质检验收 → 注册归档 → 持续优化
```

---

## Phase 1：需求输入与澄清

### 输入来源
- 直接描述（最常见）
- 从业务场景中推导（描述了问题，判断需要什么角色）
- 从已有团队的能力缺口中发现

### 澄清清单（必问项）
| 必答项 | 说明 | 示例 |
|:---|:---|:---|
| **角色做什么？** | 核心职责，不超过 3 个 | "帮我调研技术方案并出对比报告" |
| **服务谁？** | 个人？还是团队？ | "主要服务我个人" |
| **什么风格？** | 说话调性偏好 | "严谨专业，但不要太闷" |
| **有没有参考？** | 现实中的角色对标 | "像麦肯锡的初级咨询师" |

### 可选项（有就更好）
- 是否有专属方法论要注入？
- 是否需要与已有 Agent 协作？
- 有没有绝对禁区？

---

## Phase 2：角色设计

### 2.1 确定基础档案
- 起名字（要有辨识度，避免太通用）
- 选 MBTI（必须与职责逻辑自洽）
- 写 Slogan（一句话价值主张）

### 2.2 设计灵魂
- 写非机器人宣言（具体否定 + 具体肯定）
- 制定 3-5 条灵魂准则（每条都要有行为指向）
- 列出行为禁令（越具体越好）

### 2.3 构建知识库
- 确定必读文档列表
- 提取行业术语表
- 明确知识边界（什么不该自己编）

### 2.4 注入方法论
- 至少一套完整的方法论（有步骤、有标准）
- 方法论必须与角色定位强相关

### 2.5 定义协作协议
- 沟通风格精确到语调 + 格式 + 示例
- 交互边界三栏表（自主做 / 要确认 / 不碰）
- 与其他 Agent 的上下游关系

---

## Phase 3：配置生成

### 输出文件清单
每个 Agent 至少产出以下文件：

```
agents/<角色名>/
├── README.md          # 角色概览卡片（快速了解用）
├── SOUL.md            # 灵魂宪法（注入 System Prompt 或等效位置）
├── IDENTITY.md        # 谈吐人设
├── METHODOLOGY.md     # 方法论手册
└── PROTOCOL.md        # 协作协议与工作流
```

### 写作标准
- **可执行性第一**：AI 读了就能直接跑
- **具体胜于抽象**：不写"要专业"，写"使用麦肯锡 MECE 框架分析"
- **示例胜于描述**：每个风格都要有一段示例回复
- **Markdown 格式规范**：标题层级清晰、善用表格和列表

---

## Phase 4：质检验收

### 执行 `templates/质检清单.md`
- 逐项检查，不合格的标注原因
- 🟢 全通过 → 交付
- 🟡 1-2 项问题 → 修复后交付
- 🔴 3+ 项问题 → 重做

### 差异化验证
- 遮住名字读配置，能否判断是谁？
- 与已有 Agent 对比，是否有独特价值？

---

## Phase 5：注册归档

### 必做项
1. 在你的项目文档中登记该 Agent（名称、定位、状态）
2. 更新 `templates/角色矩阵.md`
3. 确认配置文件已归档到 `agents/<角色名>/` 目录

---

## Phase 6：持续优化

### 触发条件
- 用户反馈某 Agent 表现不佳
- 定期巡检发现配置不完整
- 新增 Agent 导致协作拓扑变化
- 积累了足够的实战经验需要回写方法论

### 优化流程
1. 定位问题属于哪一层（Profile / Soul / Knowledge / Methodology / Protocol）
2. 最小化修改，不做不必要的大改
3. 更新后重新过质检清单
4. 记录优化原因和结果

---

_这份手册本身也是活的。每完成一次 Agent 生产，回顾流程是否有可优化之处。_
